# Patina Core and Audio Architecture

Patina is the machine-first language platform growing out of Creation Station.

It is **not** just a text DSL for audio. It is a layered system:

- a semantic core designed for AI generation, graph authoring, and optimization
- a domain specialization for audio and DSP
- optional human-readable syntactic sugar for people who still want to write code directly

Creation Station is the proving ground for Patina Audio, but the underlying architecture is intended to generalize into other domains over time.

## Core Idea

The human-readable syntax is not the deepest truth of the system.

The deepest truth is a semantic graph and typed execution model that can be:

- generated by AI
- assembled visually in a node editor
- described through JSON project and package objects
- projected into human-friendly syntax
- compiled into optimized runtime artifacts

That means Patina has:

- **one semantic core**
- **many front ends**
- **multiple execution strategies**

## High-Level Stack

Patina is organized into layers.

### 1. Surface Layers

These are the ways authors interact with the system:

- AI-generated structured intent
- node graph editing
- JSON manifests and graph definitions
- human syntax sugar

These front ends are not separate runtimes. They all lower into the same semantic core.

### 2. Patina Core

Patina Core is the machine-native semantic layer. It defines:

- typed values
- signal streams
- control streams
- event streams
- state cells
- modules
- graph topology
- scheduling domains
- optimization metadata

Patina Core must remain domain-extensible so audio is the first specialization, not the last.

### 3. Patina Audio

Patina Audio is the first serious domain specialization. It adds:

- oscillators
- filters
- envelopes
- modulators
- sample and buffer operations
- routing
- effect chains
- instrument graphs
- timeline-aware automation
- real-time execution constraints

### 4. Runtime and Compilation

Patina artifacts can run through different backends:

- interpreter runtime for safety, bring-up, and debugging
- optimized graph runtime for fast iteration
- LLVM JIT backend for hot DSP paths
- future offline native compilation targets

## Design Principles

Patina should be built around a few non-negotiable ideas.

### Machine-first, not syntax-first

The semantic IR is the source of truth. Text syntax is optional sugar.

### One meaning, many projections

The same patch or module should be representable as:

- AI-authored structured input
- node graph
- JSON graph object
- human-readable code

All of them must compile to the same semantic meaning.

### Real-time safety is explicit

Patina must know which parts of a graph are safe for the audio thread and which are not.

### Optimization is designed in, not bolted on

The language and runtime should expose enough structure for aggressive optimization and scheduling.

### Domain specialization over hard-coding

Audio is first, but the core should later support animation, simulation, procedural visuals, games, and other domains by adding specialized semantics rather than rebuilding from scratch.

## Patina Project Structure

Patina projects should be explicit and simple to inspect.

Suggested layout:

```text
my-project/
  patina-project.json
  patina-lock.json
  src/
  graphs/
  patches/
  modules/
  assets/
  build/
```

### `patina-project.json`

Defines:

- project identity
- dependencies
- build targets
- runtime settings
- optimization profile
- exported modules or patches

### `patina-lock.json`

Defines:

- resolved package versions
- artifact hashes
- compatible runtime versions

### `src/`

Optional human syntax sugar files.

### `graphs/`

Structured graph definitions that can be generated by AI, nodes, or tooling.

### `patches/`

Reusable instruments, effects, modulators, and graph macros.

### `modules/`

Packaged semantic building blocks intended for export and reuse.

## Package System

Patina packages should be treated as reusable creative and computational modules, not just source bundles.

Package identity should be scope-based:

- `@lagdaemon/core-audio`
- `@yoho11345/my-special-stuff`

Suggested package artifact:

- `.ppkg`

Suggested tooling:

- **Language:** `Patina`
- **Package/build tool:** `Alloy`

Example commands:

```text
alloy add @lagdaemon/core-audio
alloy add @yoho11345/my-special-stuff@0.4.2
alloy add ./my-special-stuff.ppkg
alloy build
alloy run
alloy publish
```

## Semantic Core Model

Patina Core should model a graph of typed computation.

The fundamental semantic categories should include:

- **Value** - scalar or structured data
- **Signal** - continuous sampled stream
- **Control** - lower-rate continuous modulation
- **Event** - discrete timestamped occurrence
- **State** - mutable storage cell with explicit ownership and lifetime
- **Resource** - external dependency such as buffers, sample assets, or host endpoints
- **Module** - reusable unit with typed inputs, outputs, and metadata
- **Graph** - connected network of modules and edges

This gives us a foundation that audio can specialize.

## Type System Direction

The type system should be semantic, not just low-level numeric typing.

Examples:

- `audio<f32, 2>`
- `control<f32>`
- `event<midi>`
- `trigger`
- `note_stream`
- `buffer<f32>`
- `module<audio<f32,2> -> audio<f32,2>>`

The type system should eventually encode:

- channel shape
- sample representation
- scheduling domain
- thread safety class
- latency expectations

## Scheduling Domains

Patina runtime should understand execution domains directly.

At minimum:

- **audio domain** - real-time safe, no blocking, no allocation
- **control domain** - parameter updates and modulation scheduling
- **event domain** - note events, transport events, triggers
- **UI domain** - view state and editor interactions
- **worker domain** - file I/O, network I/O, asset preparation, analysis

Graphs that cross domains should require explicit boundary nodes or runtime bridges.

## Patina Audio

Patina Audio is the first major specialization built on Patina Core.

It should support:

- oscillator banks
- sample playback
- envelopes
- filters
- saturators
- delays and reverbs
- modulators and automation
- mixers and sends
- effect chains
- instrument voice graphs
- timeline and transport-aware behavior

Audio-specific semantics should remain an extension over core graph semantics.

## Intermediate Representations

Patina should have more than one internal representation.

### Surface IR

A normalized representation of source syntax, JSON graph objects, or node edits.

### Semantic IR

The source of truth. Typed, validated, domain-aware, and suitable for optimization.

### Lowered Runtime IR

A scheduling- and execution-oriented form that can drive:

- interpreter runtime
- optimized runtime graph engine
- LLVM lowering

### Artifact Form

A packaged runtime object loaded by the Patina runtime.

Suggested extension:

- `.pta` = Patina artifact

The artifact should contain:

- version and ABI metadata
- lowered graph or executable sections
- dependency references
- runtime requirements
- exported entry points
- optional debug metadata

## Compilation Pipeline

The compilation model should be staged.

### 1. Import or Parse

Input may come from:

- AI-generated structured definitions
- node editor graph data
- JSON graph objects
- human-readable Patina syntax

### 2. Normalize

Convert all front ends into a shared surface form.

### 3. Lower to Semantic IR

Create the typed graph model that becomes the source of truth.

### 4. Validate

Check:

- type correctness
- routing correctness
- domain boundaries
- real-time safety rules
- resource requirements

### 5. Optimize

Run configurable optimization passes.

### 6. Lower to Runtime IR

Prepare the graph for either interpreted or JIT execution.

### 7. Emit Artifact

Produce a `.pta` artifact for loading by the runtime.

## Optimization Strategy

Optimization should be configurable by profile.

Suggested profiles:

- `debug`
- `interactive`
- `realtime`
- `render`
- `size`
- `aggressive`

Suggested optimization passes:

- constant folding
- dead node elimination
- graph flattening
- common subgraph elimination
- inlining of tiny modules
- control/audio rate separation
- buffer reuse planning
- channel specialization
- block-size specialization
- SIMD-friendly lowering
- latency-aware scheduling

Not every build target needs every optimization. The system should support profile-driven compilation.

## Runtime Execution Model

Patina runtime must protect the audio thread.

That means audio-thread code must not:

- allocate memory
- block
- hit the filesystem
- hit the network
- perform unsafe host operations

Every module or node should eventually carry effect metadata such as:

- `realtime-safe`
- `deferred`
- `io-bound`
- `ui-bound`

The runtime loader should refuse invalid placements in strict modes.

## Interpreter and LLVM JIT

Patina should support two major execution paths early on.

### Interpreter

Used for:

- bring-up
- testing
- debugging
- safe early implementation

Benefits:

- easier to build first
- easier to inspect
- easier to validate semantics

### LLVM JIT

Used for:

- hot DSP paths
- specialized instrument and effect kernels
- real-time optimized execution

LLVM should sit behind the Patina semantic IR. It is a backend, not the language itself.

## Front-End Model

Patina should allow multiple authoring styles without splitting the platform.

### AI Front End

AI should be able to generate structured graphs and project objects directly.

### Node Front End

Graph editing should manipulate the same underlying semantic model.

### Human Syntax Sugar

Human-authored code should compile down to the same graph semantics.

This is important both for usability and for future research papers:

- one system
- many authoring projections
- one semantic truth

## Generalization Beyond Audio

Patina Core is intended to generalize.

Future domain specializations may include:

- `Patina Motion`
- `Patina Visual`
- `Patina Play`
- `Patina Sim`

These domains should reuse:

- typed graph semantics
- scheduling model
- package system
- artifact loader
- optimization infrastructure

Creation Station is the first proving ground, not the final scope.

## Immediate Build Direction

The near-term implementation should go deep in architecture but narrow in first scope.

### Build now

- semantic IR for audio graphs
- project and package manifests
- runtime artifact design
- validator
- interpreter
- basic optimization pipeline
- first LLVM JIT backend path for DSP hot spots

### Delay for later

- non-audio domain specializations
- rich human syntax
- advanced cross-domain tooling
- broad package ecosystem support

That keeps the platform serious without turning the first milestone into endless research.

## Relationship to Creation Station

Creation Station is the first flagship application built on this stack.

It should eventually use Patina for:

- Signal Lab patches
- effect graphs
- instruments
- automation logic
- plugin runtime artifacts
- AI-generated creative structures

That gives Creation Station a coherent language and runtime story rather than a pile of disconnected features.

## Summary

Patina should be built as:

- a machine-first semantic platform
- with audio as the first specialization
- with human syntax as optional sugar
- with AI, node graphs, and text all feeding the same core
- with interpreter and LLVM JIT backends behind a typed semantic IR

That gives us a path to ship a serious AI-native audio platform now while laying the foundation for a broader machine-native language ecosystem later.
